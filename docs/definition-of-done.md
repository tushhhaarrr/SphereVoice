# SphereVoice — Definition of Done (Global)

**Last Updated:** 2026-03-04  
**Applies To:** All phases, all code changes

---

## Code Changes

Every code change (PR) must satisfy ALL of the following before merge:

### 1. Code Quality
- [ ] Code merged to `dev` branch with **≥1 approval** from a team member
- [ ] No new lint errors: `black --check` + `ruff check` (Python), `eslint` (TypeScript)
- [ ] No new type errors: mypy (Python), `tsc --noEmit` (TypeScript)
- [ ] No `TODO`, `FIXME`, `HACK`, or placeholder comments left in shipped code
- [ ] No `Any` types (Python) or `any` types (TypeScript) introduced
- [ ] Follows module boundary rules (verified by `import-linter`)

### 2. Testing
- [ ] Integration tests pass in CI for affected module(s)
- [ ] Unit tests for pure business logic (service layer calculations, validators)
- [ ] No regressions — existing tests still pass
- [ ] Test coverage does not decrease (measured by `pytest-cov` / Vitest coverage)

### 3. Observability
- [ ] No new Sentry errors in staging for **24 hours** after deploy
- [ ] Relevant OpenTelemetry **metrics and traces emitting** for new features
- [ ] Structured logs include `tenant_id`, `trace_id`, `user_id` for request-scoped operations

### 4. API Contract
- [ ] OpenAPI spec updated if API endpoints changed (auto-generated by FastAPI)
- [ ] Pydantic schemas match Tech PRD API contracts
- [ ] Error responses follow the standard error format:
  ```json
  {
    "error": {
      "code": "ERROR_CODE",
      "message": "Human-readable message",
      "details": []
    }
  }
  ```

### 5. Documentation
- [ ] README or runbook updated if infrastructure changed
- [ ] ADR created if a non-obvious architectural decision was made
- [ ] Inline code comments for complex business logic (not obvious code)

### 6. Security
- [ ] Tenant-scoped data uses `tenant_id` column + RLS policy
- [ ] API keys encrypted with AES-256-GCM before storage
- [ ] No secrets in code, logs, or error messages
- [ ] Auth/RBAC enforced on new endpoints

---

## Phase-Level Gate

Each phase must satisfy ALL of the following before proceeding to the next:

1. All phase tasks completed per the execution plan
2. All gate check tests passing
3. Code committed and pushed to GitHub
4. No blocking issues or regressions
5. Phase deliverables verified against Tech PRD specifications

---

## Exceptions

If a DoD item cannot be satisfied, it must be:
1. Documented as a known gap with justification
2. Tracked as a follow-up task
3. Approved by the team lead
4. Resolved before the next phase gate (not deferred indefinitely)
